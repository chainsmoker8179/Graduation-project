# 周报生成 Skill 设计

## 1. 目标

构建一个全局可复用的 `weekly-report-writer` skill，用于根据当前对话、指定聊天记录或指定实验目录，自动生成简洁中文周报，并将结果写入 `/mnt/d/毕业设计/周报/` 目录下的 Markdown 文件，文件名沿用现有 `周报XX-XX-XX.md` 的命名风格。

## 2. 使用场景

本 skill 需要覆盖三类输入来源：

1. 当前对话
   - 直接基于当前会话上下文提取本周工作内容、问题、排查与结果。
2. 指定聊天记录
   - 用户提供一个整理后的聊天记录文件、会议记录文件或文本摘要，skill 从中抽取周报信息。
3. 指定实验目录
   - 用户提供一个实验目录或项目目录，skill 通过读取其中的 `reports/`、`docs/`、关键脚本和结果文件，整理周报内容。

## 3. 输出要求

默认输出到：

- `/mnt/d/毕业设计/周报/`

默认输出为 `.md` 文件，并自动推断下一个文件名。命名逻辑需要兼容当前目录中的：

- `周报01-1-18.md`
- `周报02-01-25.pdf`
- `周报03-01-31.pdf`

也就是说，下一份 Markdown 默认应命名为：

- `周报04-<当次日期>.md`

其中编号在现有周报中递增，日期部分采用 `MM-DD` 格式。

## 4. 内容结构

周报正文默认采用固定模板，强调简洁、具体、可追溯。正文至少包含以下部分：

1. 本周工作内容
2. 遇到的问题
3. 排查与解决过程
4. 当前结果与产出
5. 下周计划

写作约束：

- 默认中文
- 只写事实性内容，不写空泛表态
- 明确写“做了什么、出现了什么问题、如何定位、如何解决、结果如何”
- 当证据不足时，使用“待补充”而不是编造

## 5. 技术设计

skill 采用“说明文档 + 小脚本辅助”的结构：

1. `SKILL.md`
   - 定义触发条件、信息抽取流程、三种输入模式、写作模板与写作禁忌。
2. `scripts/next_report_name.py`
   - 负责扫描 `/mnt/d/毕业设计/周报/`，根据已有 `周报*.md` / `周报*.pdf` 自动推断下一个文件名。
3. `scripts/init_weekly_report.py`
   - 负责在目标目录创建新的 Markdown 文件并写入周报骨架。
4. `references/template.md`
   - 提供固定周报结构模板，供 skill 使用。

skill 本身不负责“凭空生成事实”。它只负责：

- 从当前对话或指定资料中抽取事实
- 按模板归纳
- 调用脚本生成目标文件
- 将内容写入目标 `.md`

## 6. 行为边界

本 skill 需要遵守以下边界：

1. 不伪造工作内容、实验结果或问题定位结论。
2. 如果用户未提供足够的历史记录或目录内容，只输出当前可确认的信息。
3. 如果目录中只有结果文件而缺少排查过程，问题与解决过程部分应保守表述。
4. 由于目标输出目录位于 `/mnt/d/毕业设计/周报/`，skill 需要在必要时通过带说明的提升权限命令完成落盘。

## 7. 验证方式

最小验证包括三项：

1. skill 目录结构通过 `quick_validate.py` 校验。
2. `next_report_name.py` 能在当前周报目录中生成正确的下一个文件名。
3. 使用“当前对话”模式生成一份 Markdown 周报，并确认：
   - 文件创建成功
   - 文件名符合周报命名规则
   - 正文包含五个既定部分
