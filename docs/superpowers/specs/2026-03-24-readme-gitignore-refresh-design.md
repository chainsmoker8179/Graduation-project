# README 与 .gitignore 补充设计

## 背景

当前仓库已经形成较完整的研究与实验体系，但根目录仍存在两个明显问题：

1. 缺少根目录 `README.md`，导致仓库首次访问者无法快速理解项目目标、目录结构、关键脚本与文档入口。
2. 现有根 `.gitignore` 仅覆盖了少量缓存与部分实验产物，对本地常见临时文件、notebook 检查点和日志文件的覆盖仍不完整。

与此同时，仓库中已经存在大量被纳入版本管理的实验报告与结果目录，因此 `.gitignore` 不能采用“激进忽略所有实验输出”的策略，否则会与当前仓库跟踪口径冲突。

## 目标

本轮工作只做两件事：

1. 新增一个中文根目录 `README.md`，兼顾项目简介、最小复现入口与文档索引。
2. 以保守策略补充根 `.gitignore`，只忽略明确的本地缓存、编辑器临时文件和 notebook 检查点，不影响当前已跟踪实验结果。

## 用户确认后的约束

用户已确认：

- `README` 以中文为主；
- 风格采用折中版，而不是极简版或重型复现实验手册；
- `.gitignore` 采用保守策略，不扩大到大范围忽略实验目录；
- 重点是提高仓库可读性与可维护性，而不是重构项目结构。

## 方案选择

### 方案 A：折中版 README + 保守 .gitignore

`README.md` 覆盖以下内容：

- 项目简介
- 当前实验主线
- 核心目录结构
- 环境与依赖说明
- 常用脚本入口
- 文档索引入口
- 本地资产说明

`.gitignore` 仅补充：

- `.ipynb_checkpoints/`
- `*.log`
- `*.tmp`
- `*.swp`
- `.DS_Store`

优点：

- 风险最低；
- 与当前仓库状态一致；
- 不会误伤已纳入版本控制的报告与结果目录；
- 足以满足 GitHub 公开访问和后续自我维护需要。

缺点：

- 不会自动清理所有潜在大体积实验输出；
- 仍然保留一定程度的人工管理成本。

### 方案 B：重型 README + 激进 .gitignore

把 `README` 写成完整复现实验手册，同时大范围忽略各类实验输出目录。

优点：

- 对新读者最友好；
- 理论上更整洁。

缺点：

- 与当前仓库已跟踪的实验结果策略冲突；
- 更容易误伤现有研究产物；
- 成本高且超出本轮目标。

## 推荐方案

采用方案 A。

## 设计细节

### README 结构

根目录 `README.md` 建议采用以下结构：

1. 项目概述
2. 当前已完成的实验主线
3. 目录结构说明
4. 环境与运行说明
5. 常用脚本入口
6. 核心文档索引
7. 注意事项

其中“核心文档索引”不重复展开所有实验细节，而是直接指向：

- `reports/项目实验总览与文档索引.md`

### .gitignore 调整边界

本轮不会：

- 忽略整个 `reports/`
- 忽略整个 `origin_model_pred/`
- 忽略当前已经被提交的 CSV / JSON / Markdown 结果文件

本轮只补充明确的临时文件与本地缓存规则。

## 预期结果

完成后，仓库将具备：

1. 一个能解释“项目做什么、从哪开始读、怎么找到关键脚本”的根目录 `README.md`；
2. 一份更稳妥的根 `.gitignore`，减少临时文件误入仓库；
3. 不改变现有实验结果跟踪策略，不影响当前 `master` 的继续实验能力。
