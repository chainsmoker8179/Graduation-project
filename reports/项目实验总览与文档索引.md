## 3. 实验主线的逻辑先后关系

如果把当前项目压缩成一条主线，它的逻辑顺序可以概括为：

1. 先解决“Alpha158 因子模块为什么不可导、应该怎样可导化”的问题。
2. 再解决“BPDA 这种前向硬算子、反向软近似的方法，在算子级别是否可信”的问题。
3. 然后解决“即便单个算子可导，整条 `OHLCV -> 因子 -> loss` 的端到端梯度链路是否真的能打通”的问题。
4. 在端到端梯度链路确认成立之后，再进一步解决“这些梯度能否服务于真实已训练基线模型的白盒攻击”的问题。
5. 进入白盒攻击阶段后，先通过小样本 smoke 证明攻击链路能跑，再通过扩大样本和收紧 clean gate 检查这条链路是否足够接近旧版 Qlib 流水线。
6. 当扩大样本后发现 clean 对齐偏弱时，再做根因定位，最终发现问题主要不是公式本身，而是模型真实输入的 20 维特征顺序错位。
7. 修正顺序错位后重导资产并正式重跑攻击，形成当前较可靠的 `expanded_v6` 基线。

也就是说，整个项目并不是一开始就在做 FGSM/PGD，而是经历了下面这条逐步加深的问题链：

`可导化设计 -> BPDA 局部验证 -> 端到端梯度验证 -> 原始 OHLCV 攻击链路设计 -> 小样本攻击 smoke -> 扩大样本与 clean gate 收紧 -> clean 偏弱根因定位 -> 正式重跑攻击`

在此基础上，当前主线又向前延伸了一步，即从“样本级预测误差攻击”继续推进到“部分股票攻击下的组合级回测退化验证”。新的阶段顺序可以写成：

`可导化设计 -> BPDA 局部验证 -> 端到端梯度验证 -> 原始 OHLCV 攻击链路设计 -> 小样本攻击 smoke -> 扩大样本与 clean gate 收紧 -> clean 偏弱根因定位 -> 正式重跑攻击 -> 每日随机 5% 股票的组合级 partial attack backtest -> 5% 多随机种子稳定性验证 -> 1%/5%/10% 攻击比例对比`

## 4. 各实验的作用、结论与衔接关系

### 4.1 阶段 A：可导特征模块建设

对应文档：

- `可导特征模块章节.md`
- `plan.md`

这一阶段回答的问题是：为什么传统 Alpha158 因子表达式会阻断梯度，以及如果希望以后做端到端训练或梯度型攻击，应该如何改造因子模块。该阶段的产出不是具体实验结论，而是方法框架和工程推进顺序。核心思想是前向保持 Qlib/Alpha158 的硬语义一致，反向通过 BPDA 或 soft-rank 等平滑近似恢复梯度。

这一阶段的作用相当于整个项目的“理论与工程底座”。没有这一步，后续无论是端到端训练验证还是白盒攻击，都没有可靠的可微因子图可用。

这一阶段仍然存在的问题是：它更偏设计与实施计划，尚未回答“这些近似到底是否真的可信”。因此下一步自然进入 BPDA 算子级验证。

### 4.2 阶段 B：BPDA 算子级梯度验证

对应文档：

- `scripts/BPDA_GRAD_VALIDATION.md`
- `reports/validate_bpda_gradients_中文详解报告.md`

这一阶段回答的问题是：对 `max/min`、`greater/less`、`quantile`、`idxmax/idxmin`、`rank` 这类离散或非光滑算子，BPDA 是否同时满足两件事，即前向仍保持硬语义一致，反向又能给出和软替代近似一致的梯度。

这一阶段的结论是：BPDA 验证脚本已经从三个层面建立了局部可信性。

1. 前向与硬算子对齐。
2. BPDA 反向梯度与软替代算子的梯度对齐。
3. 梯度方向性、边界敏感性、单调性等语义符合预期。

这一阶段表明，局部算子级别的“可导替代”是成立的，但仍有一个未解决问题：单个算子可导，并不自动推出真实市场数据上的整条因子流水线可把监督损失传回原始输入。于是后续推进到端到端梯度验证。

### 4.3 阶段 C：端到端梯度回传验证

对应文档：

- `reports/e2e_grad_smoke/e2e_grad_validation_report.md`
- `reports/e2e_grad/e2e_grad_validation_report.md`
- `reports/e2e_grad_validation.md`

这一阶段回答的问题是：在真实 Qlib 数据上，从监督损失出发，梯度是否真的能够跨过完整的可微因子提取器，稳定回传到原始 OHLCV 输入。

这一阶段的实验分成三层：

1. `full_e2e`：检查完整 157 个 OHLCV 可计算因子是否能支持端到端梯度回传。
2. `subset_e2e`：聚焦 19 个离散算子相关因子，检查这些更困难的因子子集是否仍能回传梯度。
3. `factor_probe`：对单个因子施加探针损失，定位到底是哪一类因子或哪一列输入存在弱梯度。

这一阶段的主要结论是：

- 完整 157 因子配置与离散子集配置的 batch 通过率都为 `1.0`。
- 说明“整体梯度通路成立”这一点已经被真实数据验证。
- 但在离散子集下，`volume` 列表现出持续弱响应。
- 进一步 probe 后，弱梯度主要集中在 `VSUMP5`、`VSUMN5`、`VSUMD5` 这三个基于成交量差分的比较因子上。

因此，这一阶段给出的不是“所有分支都同样健康”的结论，而是“整条链路已经打通，但局部 volume-diff 分支仍有弱梯度区域”。这为后续工作提供了两个重要前提：第一，可以继续往下推进基于梯度的攻击；第二，后续要对局部弱梯度分支保持警惕。

### 4.4 阶段 D：原始 OHLCV 白盒攻击链路设计

对应文档：

- `docs/superpowers/specs/2026-03-11-raw-ohlcv-lstm-whitebox-attack-design.md`
- `docs/superpowers/plans/2026-03-11-raw-ohlcv-lstm-whitebox-attack.md`

这一阶段回答的问题是：既然可微因子流水线已经可以回传梯度，那么如何把它嫁接到历史上已经训练好的 Qlib LSTM 基线模型上，从而在原始 OHLCV 空间中实施 FGSM/PGD。

这里的关键难点不在攻击算法本身，而在链路重建。旧版 LSTM 并不直接消费原始 OHLCV，而是消费 `FilterCol + RobustZScoreNorm + Fillna` 处理后的 20 个 Alpha158 特征。因此，这一阶段的设计工作实际上是在回答：如何在不重训旧模型的前提下，搭建一条可微且尽量等价的

`原始 OHLCV -> 20 个旧版特征 -> Torch 版 RobustZScoreNorm/Fillna -> 旧版 LSTM`

前向与反向路径。

这一阶段的作用是把前面的“可导因子能力”真正转化为“对白盒攻击有用的工程链路”。但设计完成后，还没有实验层面的证据，所以需要先做小样本 smoke attack。

### 4.5 阶段 E：第一版小样本白盒攻击 smoke

对应文档：

- `reports/lstm_whitebox_attack_smoke_v1/attack_report.md`
- `reports/lstm_whitebox_attack_论文段落.md`

这一阶段回答的问题是：在极小规模、可人工核对的匹配样本子集上，原始 OHLCV 攻击链路是否真的可运行，FGSM 和 PGD 是否至少能够提升损失。

这一阶段的主要结果是：

- 只有 7 个样本，但 clean gate 可以通过，说明梯度能回传到原始 OHLCV。
- `clean_loss = 0.00303747`
- `fgsm_loss = 0.03213169`
- `pgd_loss = 0.05664916`
- `clean Spearman = 0.428571...`

这一步的作用是“证明链路可跑通”，而不是形成稳定统计结论。它已经表明 FGSM/PGD 能基于当前梯度链路放大预测误差，但也暴露出一个明显问题：clean 对齐还不够强，`Spearman` 只有约 `0.43`，不足以支撑更大规模结论。因此，后续增加了更严格的 clean gate，并扩大样本规模。

### 4.6 阶段 F：扩大样本并收紧 clean gate

对应文档：

- `docs/superpowers/plans/2026-03-11-attack-clean-gate-and-expanded-smoke.md`
- `reports/lstm_whitebox_attack_expanded_probe_v1/attack_report.md`
- `reports/lstm_whitebox_attack_expanded_v2/attack_report.md`
- `reports/lstm_whitebox_attack_expanded_v3/attack_report.md`
- `reports/lstm_whitebox_attack_expanded_report.md` 第 1-7 节

这一阶段回答的问题是：在样本数量扩大后，如果同时加入更严格的特征级对齐指标，当前攻击链路是否仍然成立，FGSM/PGD 是否仍能稳定起作用。

这一阶段的主要改动是：

1. clean gate 不再只检查“梯度是否存在”，而是加入 `feature_mae`、`feature_rmse`、`feature_max_abs` 等中间特征对齐指标。
2. 离线导出侧新增 `matched_feature_windows.pt`，直接导出 Qlib 侧真实输入窗口，方便与 Torch 在线桥接结果逐项比较。
3. 将样本从极小规模扩大到 `62` 个有效样本。

这一阶段的主要结果是：

- `clean_loss = 0.01622363`
- `spearman_to_reference = 0.10035`
- `feature_mae_to_reference = 0.03452`
- `feature_rmse_to_reference = 0.09824`
- `fgsm_loss = 0.07544520`
- `pgd_loss = 0.10530549`

这一步非常关键，因为它暴露出一个更深的问题：中间特征误差其实不算大，但最终 `pred.pkl` 对齐却异常偏弱，`Spearman` 只有 `0.10035`。这说明问题不像“整个特征公式都错了”那么简单，更像是某种更隐蔽的语义错位。于是，下一步转向 clean 偏弱根因定位。

### 4.7 阶段 G：clean 偏弱根因定位

对应文档：

- `reports/lstm_whitebox_attack_expanded_report.md` 第 8-9 节
- `reports/lstm_whitebox_attack_expanded_v5_quick/attack_report.md`

这一阶段回答的问题是：为什么 Torch 在线重建特征与 Qlib 参考特征在 MAE/RMSE 上看起来并不夸张，但最终模型输出的排序对齐却很差。

最终定位出的主因是：20 维特征顺序错位。

具体来说：
1. 旧 notebook 虽然写了一个 `feature_cols` 列表，但 Qlib 的 `FilterCol` 只筛列，不会按这个列表重新排列。
2. 训练时模型真正看到的输入顺序，其实是 Alpha158 原始列顺序经过 `FilterCol` 保留后的顺序。
3. 攻击链路此前把手写的 `feature_cols` 顺序误当成真实模型输入顺序，因此出现了“特征张量本身看起来差不多，但输入维度语义错位”的问题。

这一定位非常重要，它说明此前 `v2/v3` 的 clean 偏弱不是因为模型权重错误、预测文件错误，也不是特征公式整体错误，而是一个更隐蔽的维度排列问题。

`v5_quick` 的作用是快速确认根因判断是否正确。它的结果显示，只要把特征顺序和统计量顺序一起修正，clean `Spearman` 就会明显上升，这为后续正式重导资产和重跑攻击提供了依据。

### 4.8 阶段 H：修正后的正式重跑攻击

对应文档：

- `reports/lstm_whitebox_attack_expanded_v6/attack_report.md`
- `reports/lstm_whitebox_attack_expanded_report.md` 第 8-9 节

这一阶段回答的问题是：在修复 20 维特征顺序错位之后，攻击链路的 clean 对齐是否显著提升，FGSM/PGD 的攻击效应是否也同步增强。

这一阶段的正式结果是：

- 样本数：`62`
- `clean_loss = 0.01431117`
- `spearman_to_reference = 0.7513031653697968`
- `fgsm_loss = 0.10467355`
- `pgd_loss = 0.28165913`
- `fgsm_mean_abs_pred_shift = 0.19706690`
- `pgd_mean_abs_pred_shift = 0.36676064`

与修正前的 `v2/v3` 相比，这一步给出了两个更可靠的结论：

1. clean 偏弱的主导根因已经被定位并修复，`Spearman` 从 `0.10035` 提升到 `0.75130`。
2. clean 对齐提升之后，FGSM 和 PGD 的攻击效果也更强，说明攻击图与旧版模型真实输入空间的一致性明显提高。

因此，截至目前，`expanded_v6` 应该被视为原始 OHLCV LSTM 白盒攻击实验的当前正式基线，而 `v2/v3` 主要用于保存“问题是如何暴露和定位出来的”这段过程证据。

### 4.9 阶段 I：部分股票攻击下的组合级回测验证

对应文档：

- `reports/partial_attack_backtest_seed0_ratio5_v1_实验报告.md`
- `reports/partial_attack_backtest_seed0_ratio5_v1/README.md`

这一阶段回答的问题是：即使不攻击全市场股票，只在每个交易日随机攻击其中固定比例的一小部分股票，白盒扰动是否仍会传导到组合构建和回测层面，并表现为收益下降、回撤上升和排序能力减弱。

这一阶段采用了四组并行比较：`reference_clean` 表示原始 `pred.pkl` 回测基线；`partial_clean` 只替换被抽中股票的 clean 前向分数，用于隔离“局部替换本身”的影响；`partial_fgsm` 和 `partial_pgd` 则在相同股票子集上分别替换为 FGSM 和 PGD 的攻击分数。当前固定口径为完整测试区间、每日随机 `5%` 股票、`seed=0`，并以 `partial_clean` 作为主对照。

这一阶段的主要结果是：

- 候选攻击样本数为 `52,422`，其中 `50,874` 条样本成功对齐到可攻击资产，且全部通过 clean gate。
- `partial_clean - reference_clean` 很小：年化超额收益仅变化 `+0.00477`，最大回撤仅变化 `-0.00092`，`rank_ic_mean` 仅变化 `+0.00004`。
- `partial_fgsm - partial_clean` 明显退化：年化超额收益下降 `0.28002`，最大回撤恶化 `0.05001`，`rank_ic_mean` 下降 `0.01299`。
- `partial_pgd - partial_clean` 退化更强：年化超额收益下降 `0.39832`，最大回撤恶化 `0.06058`，`rank_ic_mean` 下降 `0.01438`。

这一步的作用非常直接。它把此前“攻击能放大单样本预测误差”的结论，推进为“局部股票攻击已经足以在组合级回测中造成明确退化”的结论。与此同时，`partial_clean` 与 `reference_clean` 的偏差仍然较小，因此可以较有把握地把收益和风险指标的变化归因于攻击扰动本身，而不是归因于回测拼接方式。到这一阶段为止，项目已经从可导性验证走到了组合级攻击有效性验证。后续如果要进一步强化论文证据，最自然的方向就是做多随机种子、不同攻击比例和更严格的 feature-level clean gate 复核。

### 4.10 阶段 J：部分股票攻击的多随机种子与比例扩展验证

对应文档：

- `reports/partial_attack_backtest_multiseed_ratio5_union/多随机种子稳定性报告.md`
- `reports/partial_attack_ratio_sweep_multiseed/攻击比例对比报告.md`
- `docs/superpowers/specs/2026-03-13-partial-attack-ratio-sweep-design.md`
- `docs/superpowers/plans/2026-03-13-partial-attack-ratio-sweep.md`

这一阶段回答的问题是：前一阶段在单个随机种子、固定 `5%` 攻击比例下观察到的组合退化，究竟是偶然现象，还是在更稳定的随机抽样和更广的攻击覆盖率范围内都能复现。

这一阶段分成两步：

1. 先在共享资产口径下，对每日随机攻击 `5%` 股票的 partial backtest 做 `seed=0..4` 复跑，检查组合级退化是否具有跨种子稳定性。
2. 再把攻击比例扩展到 `1% / 5% / 10%`，比较比例提升是否会系统性放大组合退化。

这一阶段的主要结论是：

- 在 `5%` 比例下，FGSM 和 PGD 相对 `partial_clean` 的四项主指标在五个种子上全部为负，说明组合级退化不是单次抽样偶然。
- 当比例从 `1%` 提高到 `5%` 再提高到 `10%` 时，FGSM 与 PGD 在年化超额收益、最大回撤、`RankIC` 和信息比率四个维度上的平均退化都呈现单调或近似单调增强。
- PGD 在收益类与排序类指标上持续强于 FGSM，说明更强的迭代型攻击会在组合级层面带来更重的退化。

这一阶段的意义在于，它把“局部攻击有效”从单次示例推进成了“跨随机种子稳定”以及“随攻击覆盖率增强而系统性变差”的更强证据。这使得 partial attack backtest 不再只是一个存在性验证，而开始具备趋势性和稳健性结论。

## 6. 截至目前的项目状态总结

截至目前，可以把项目进展概括为以下几点：

1. 可导 Alpha158 模块的核心方法论已经建立，BPDA 与相关平滑近似不再停留在概念层，而是已经被组织进可执行、可验证的工程方案。
2. BPDA 在离散算子级别的局部验证已经完成，说明“前向硬语义一致、反向软近似给梯度”的基本策略是站得住的。
3. 在真实 Qlib 数据上，完整 `OHLCV -> 因子 -> loss` 的端到端梯度回传已经被验证是成立的；但 `VSUMP5 / VSUMN5 / VSUMD5` 这类 volume-diff 分支仍表现出局部弱梯度问题。
4. 面向历史已训练 LSTM 基线模型的原始 OHLCV 白盒攻击链路已经搭建完成，不需要先重新训练模型，就可以在原始输入空间执行 FGSM/PGD。
5. 攻击链路经历了“小样本 smoke -> 扩大样本 -> clean 偏弱定位 -> 特征顺序修正 -> 正式重跑 -> 部分股票攻击组合回测 -> 多随机种子稳定性验证 -> 攻击比例对比”这条完整闭环，当前既有样本级攻击基线 `expanded_v6`，也有组合级 `partial attack backtest` 的稳定性与比例趋势证据。

同时，当前项目仍然保留三个明显的后续方向：

1. 对 `VSUMP5 / VSUMN5 / VSUMD5` 的弱梯度现象做更有针对性的边界样本验证，确认问题究竟来自 BPDA 近似本身还是随机 batch 覆盖不足。
2. 在当前 `partial attack backtest` 已经完成多随机种子和不同比例验证的基础上，继续做多次随机种子复跑、显著性检验以及更细的收益归因/换手分解，把“稳定退化”进一步推进为更完整的金融实验结论。
3. 将当前原始 OHLCV 攻击链路从 LSTM 扩展到其他已训练基线模型，例如 transformer，从而验证攻击方法是否具有跨模型一致性。

## 7. 一句话总结

当前项目的实验逻辑已经形成清晰闭环：先证明可导因子模块在算子级别和端到端级别都能稳定回传梯度，再把这条梯度链路接到历史已训练的 LSTM 基线上实施原始 OHLCV 白盒攻击，随后修正 clean 偏弱问题并得到 `expanded_v6` 基线，最后进一步证明局部股票攻击不仅会传导到组合级回测退化，而且这种退化还会在多随机种子下稳定出现，并随攻击覆盖率上升而系统性增强。
